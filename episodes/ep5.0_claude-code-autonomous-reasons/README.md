# ep5.0 Claude Codeが自律的に動けた理由

この回では、Claude Code が「指示なし」で動いた3つの仕掛けを解説しました。

---

## この回でやったこと

1. CLAUDE.md が「最初に読む場所を指すポインタ」として機能している仕組みを見せる
2. スキル（`/implement`）が手順をまるごと1コマンドにまとめている仕組みを見せる
3. 仕様書の未確定事項（`spec §8`）が FP の回答待ちで実装されなかった理由を見せる

---

## 3つの仕掛け

### 仕掛け1: CLAUDE.md はポインタ

```markdown
<!-- CLAUDE.md の例 -->
詳細な作業手順は docs/rules/00_workflow.md を参照する。
```

ルールを全部 CLAUDE.md に書くのではなく、「ここを読め」と書く。
Claude Code が起動時に読む場所を1つに絞ることで、最新のルールが常に参照される。

### 仕掛け2: スキルで手順を1コマンドに

```bash
/implement
```

「テストを書いて → 設計して → 実装する」という手順を1つのスキルにまとめる。
目的（何を達成するか）だけを渡せば、手順は Claude Code が知っている。

### 仕掛け3: 未確定事項は実装しない

```markdown
<!-- spec §8 の例 -->
育休中の収入モデル: FP確認が必要（未確定）
```

仕様書に「FP確認が必要」と書いてあった箇所は、Claude Code が実装しなかった。
答えていないことは動かさない、という前提を書いておいたから。

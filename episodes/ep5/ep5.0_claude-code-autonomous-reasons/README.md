# ep5.0 Claude Codeが自律的に動けた理由

この回では、Claude Code が「目的だけを渡されて」動けた理由を、3つの仕掛けに分解して解説しました。
「なぜ細かく手順を伝えなくてもよかったのか」という問いへの答えです。

---

## この回でやったこと

1. CLAUDE.md が「最初に読む場所を指すポインタ」として機能している仕組みを見せる
2. スキル（`/implement`）が手順をまるごと1コマンドにまとめている仕組みを見せる
3. 仕様書の未確定事項（`spec §8`）が FP の回答待ちで実装されなかった理由を見せる

---

## 3つの仕掛け

### 仕掛け1: CLAUDE.md はポインタ

ルールを全部 CLAUDE.md に書くと、ファイルが肥大化して矛盾が生じやすくなります。
このプロジェクトでは CLAUDE.md を「どのドキュメントを読むべきか」だけを示すポインタとして使っています。

```markdown
<!-- CLAUDE.md の実際の書き方 -->
詳細な作業手順は docs/rules/00_workflow.md を参照する。
エージェント一覧・レビューフローは docs/rules/00_workflow.md の「エージェント活用」セクションを参照する。
```

Claude Code が起動時に読む場所が1つに絞られるため、ルールを更新するときは参照先ドキュメントだけ変えればよい。
CLAUDE.md 自体はほぼ変わらない。これが「コンテキストを整える」という設計の意味です。

**なぜこれで動いたか:**
Claude Code はセッション開始時に CLAUDE.md を読み、そこから先のドキュメントをたどります。
ポインタが正確であれば、最新のルールが常に届きます。
ルールを CLAUDE.md に直書きしていると、コピーが増えて「どちらが正しいか」が曖昧になります。

---

### 仕掛け2: スキルで手順を1コマンドに

`/implement` というスキルを呼ぶだけで、以下の手順が自動的に実行されます。

```bash
# 目的だけを渡す
/implement
```

スキルの内部では「tester → architect → engineer」という順序が定義されています。

```markdown
<!-- .claude/commands/implement.md の概要 -->
1. tester エージェントにテストを書かせる
2. architect エージェントに設計を確認させる
3. engineer エージェントに実装させる
```

**なぜこれで動いたか:**
「何をするか（目的）」と「どうやるか（手順）」が分離されています。
目的だけを渡せば、手順は Claude Code がスキルから読みとります。
手順を毎回チャットに書く必要がなく、手順の変更もスキルファイルだけで完結します。

---

### 仕掛け3: 未確定事項は実装しない

仕様書 `spec §8`（育休中の収入モデル）に以下の記述がありました。

```markdown
<!-- spec §8 の実際の記述 -->
育休中の収入モデル: FP確認が必要（未確定）
```

Claude Code はこの箇所を実装しませんでした。
「FP確認が必要」という記述を読んで、実装するかどうかの判断をした結果です。

**なぜこれで動いたか:**
仕様書に「未確定」と明示されていたため、Claude Code が独自の仮定で実装して後から差し替えが必要になる、という事態を避けられました。
「答えが出ていないことは動かさない」というガードレールを、仕様書の記述として埋め込んでいたからです。

さばきゃんの「……それ、私が答えてなかったからか」は、このガードレールが実際に機能したシーンへの反応です。

## 「目的を渡す」とはどういうことか

この回全体を通じて伝えたかったのは、「細かく手順を伝えなくても動く」ための設計の話です。

- CLAUDE.md はポインタ → **コンテキストを整える**
- スキルは手順の束 → **目的だけを渡せるようにする**
- 仕様書の未確定フラグ → **判断の根拠を渡す**

どれも「何かを送り込む」のではなく、「Claude Code が判断するために必要なコンテキストを用意する」という発想で設計しています。

# clip-relative time vs scene-elapsed time

## 問題

アニメーションの時刻として `scene_elapsed`（シーン開始からの累積経過秒数）を使うと、
「長い発話の後に zoom ディレクティブが来る」シーンでアニメーションが瞬時に終わる問題が起きる。

```
## シーン4: zoom in 長いセリフ直後

<!-- speaker: yumu -->
これは長い発話のシーンです。10秒くらいかかります。  ← clip_t=0〜10

<!-- zoom: in scale=1.4 duration=1.5 -->
長い発話の後で zoom in が動くか確認します。         ← clip_t=0〜3（新しいクリップ）
```

上記の台本では、2番目のクリップは「新しいクリップ」として生成される。
`scene_elapsed` を使うと、このクリップが始まる時点で既に `scene_elapsed ≈ 10.0` になっている。

```python
# scene_elapsed を使った場合（バグ）
progress = min(10.0 / 1.5, 1.0)  # = 1.0（既にアニメーション完了）
scale = 1.0 + (1.4 - 1.0) * 1.0  # = 1.4（瞬時に最大値）
```

アニメーションが即座に完了し、「じわっとズームする」演出が消える。

## 原因

`scene_elapsed` は「シーン内の全クリップを通算した経過時間」であり、
クリップをまたぐと値が積み上がる。zoom アニメーションの進行計算に使うには不適切。

## 解決策: clip_t を使う

`clip_t`（クリップ先頭を 0.0 とした相対時刻）を使うことで、
どのクリップが来ても t=0 からアニメーションが始まる。

```python
# clip_t を使った場合（正しい）
def calc_zoom_scale(clip_t, direction, scale, duration, easing):
    raw_progress = min(clip_t / duration, 1.0)  # clip_t=0 → 0.0 から始まる
    ...
```

MoviePy のクリップの `make_frame(t)` に渡される `t` がそのまま `clip_t` として使える。

## 設計の原則

**「何に対する経過時間か」を明確にする。**

| 用途 | 使うべき時刻 | 理由 |
| ---- | ------------ | ---- |
| zoom アニメーション | `clip_t`（クリップ相対） | クリップごとに毎回リセットしたい |
| thinking ズーム | `scene_elapsed`（シーン累積） | クリップをまたいでも連続して拡大し続けたい |
| sad 沈下アニメーション | `scene_elapsed`（シーン累積） | 「ずっと悲しんでいる」表現に時間的連続性が必要 |

同じ「時刻」でも、アニメーションの意図によって使い分けが必要。

## 教訓

- アニメーション関数を実装するとき、引数名だけでなく「何を起点とした時刻か」を仕様として明示する
- `scene_elapsed` と `clip_t` は別物。関数シグネチャに両方が混在しないよう注意する
- 「長い発話の後でも動くか」を検証するテストシナリオを必ずテスト台本に含める

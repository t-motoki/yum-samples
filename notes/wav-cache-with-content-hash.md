# WAV キャッシュをコンテンツハッシュで管理する — テキスト変更を自動検知する

VOICEVOX で生成した音声（WAV）のキャッシュ設計の変遷と、
「テキストを変えても古いキャッシュが使われる」バグの解決方法を記録します。

## 問題

台本のナレーションテキストを修正しても、古い音声がそのまま使われ続ける問題が発生した。

```text
# 修正前のナレーション
「Pythonとフリーツールだけで動画を作れます」

# 修正後
「無料のOSSツールだけで動画を作れます」

→ 再生成しても古い WAV が流れる
```

## 原因: シーン名だけをキャッシュキーにしていた

```python
# 問題のある実装
cache_path = output_dir / f"{scene.id}.wav"
if cache_path.exists():
    return cache_path  # テキストが変わっても同じファイルを返してしまう
```

キャッシュキーが `scene_id`（シーン名）のみだったため、
テキスト内容が変わってもキャッシュヒットしてしまっていた。

## 解決策: テキストの SHA-256 ハッシュをキャッシュキーに含める

```python
import hashlib

def _cache_path(scene_id: str, text: str, output_dir: Path) -> Path:
    text_hash = hashlib.sha256(text.encode()).hexdigest()[:8]
    return output_dir / f"{scene_id}_{text_hash}.wav"
```

キャッシュキーを `{scene_id}_{sha256[:8]}.wav` にすることで、
テキストが1文字でも変わると別ファイル名になり、自動的に再生成される。

```text
# before
scene_01.wav

# after
scene_01_a3f7c9b2.wav  ← テキスト変更 → ハッシュ変更 → 別ファイルとして生成
scene_01_e8b2a1d4.wav
```

## ハッシュを8文字に切り詰める理由

- SHA-256 の完全な 64文字は長すぎてファイル名が読みにくい
- 8文字（32ビット）でも同じプロジェクト内での衝突確率は十分低い
- 1プロジェクトで管理するシーン数は数十〜数百程度

## 副作用: 古いキャッシュファイルが溜まる

テキストを変更するたびに古いハッシュのファイルが残る。
定期的に整理するか、エピソード完成後にクリーンアップするとよい。

```bash
# 特定エピソードの WAV を全削除して再生成
rm -rf output/<ep_id>/audio/
python generate_episode.py inputs/<ep_id>/script.md
```

## 設計のポイント

**「何がキャッシュキーか」を明示的に決める。**

キャッシュの無効化条件が曖昧なまま実装すると、このような「変えたのに変わらない」バグが起きやすい。

| キャッシュキーの候補 | 特性 |
| --- | --- |
| シーン名のみ | 高速・同じ名前なら再利用。テキスト変更を検知できない |
| テキスト全文のハッシュ | テキスト変更を確実に検知。1文字変えると全再生成 |
| シーン名 + テキストハッシュ | バランスよい。本記事の採用方式 |
| タイムスタンプ | キャッシュが効かない（毎回再生成） |

# リップシンクツール選定: 「映像出力型」か「データ出力型」か

リップシンクツールを選ぶとき、「映像を直接出力するか」「タイミングデータだけを出力するか」
の違いを最初に意識すると、既存パイプラインとの統合コストが大きく変わります。
ep3.3〜ep3.4.2 の実装で気づいた観点をまとめます。

## 2種類のリップシンクツール

### 映像出力型（Wav2Lip・SadTalker）

音声と顔画像を入力として受け取り、**口が動いた映像（MP4）を直接出力する**。

```
顔画像 + 音声 → [Wav2Lip / SadTalker] → 口が動く映像（MP4）
```

映像として完結しているため、単体で動作を確認しやすい。
一方で、出力フォーマットや解像度・フレームレートはツール側が決める。

### データ出力型（Rhubarb Lip Sync）

音声を入力として受け取り、**「この時刻にこの口の形」というタイミングデータを出力する**。
映像の合成は自分で行う。

```
音声 → [Rhubarb] → viseme タイミング JSON
                      → 自分のレンダラー → 映像（MP4）
```

## 既存パイプラインがある場合はデータ出力型が馴染む

Wav2Lip・SadTalker を既存の動画パイプラインに組み込もうとすると、
ツールが出力した映像を**さらに自分のパイプラインで合成・編集**する手順になる。

```
自分の動画パイプライン
  ├── テロップ合成
  ├── BGM 合成
  └── リップシンク映像の合成  ← ここに Wav2Lip の出力を貼り込む（座標・サイズ合わせが必要）
```

Rhubarb のようなデータ出力型は、タイミング情報を既存の表情切り替えロジックに渡すだけ。
パイプラインの構造を変えずに「口パク」機能を追加できる。

```python
# 既存の表情決定ロジックにひと手間加えるだけ
cues = extract_visemes(wav_path)  # Rhubarb を呼ぶ
expression = find_expression_at(cues, current_time)  # タイミングから表情を決める
render_frame(expression)  # 既存レンダラーで描画
```

## 比較

| 観点 | 映像出力型（Wav2Lip など） | データ出力型（Rhubarb など） |
| --- | --- | --- |
| 出力 | 映像ファイル（MP4） | タイミング JSON |
| 単体動作確認 | しやすい | タイミングデータを目視確認する |
| 既存パイプラインへの組み込み | 映像として貼り込む（座標合わせ必要） | データを渡すだけ |
| 映像品質の制御 | ツール依存 | 自分の表情絵・レンダラー次第 |
| アニメ顔への対応 | 顔検出が実写向けのため難あり | 音声解析のみのため顔の種類を問わない |

## 問いを変えると選択肢が変わる

- ❌「リップシンク映像を **作りたい**」→ Wav2Lip / SadTalker を試す
- ✅「既存パイプラインに **口パクを追加したい**」→ Rhubarb のようなデータ出力型が馴染む

「ツールが映像を出力してくれる」ことは必ずしも強みではない。
**既存の仕組みがあるなら、データだけもらって自分で制御するほうが柔軟**。

## 教訓

**ツールの「出力フォーマット」を最初に確認する。**
映像を出力するか、データを出力するかで、既存パイプラインへの組み込み方が根本的に変わる。
「デモが動いた」から「自分のシステムに組み込める」は別の話。
